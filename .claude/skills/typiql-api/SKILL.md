# Building on typiql (this app's backend data layer)

typiql is the ORM/schema layer this app's entire backend is built on
(`src-tauri/src/typiql_types.rs`, sibling repo `typiql-rs`). It generates a
complete CRUD + subscription API for every registered type, plus batched
relation resolution. **Any new data accessed via CRUD-style operations
(list/get/add/update/remove/subscribe) must be a typiql type, registered in
`typiql_schema!` — not a hand-rolled resolver, not a parallel storage
mechanism.** This mirrors how *every* existing type in this app works
(`Car`, `Dashboard`, `MonocoqueSoundDevice`, `LfeChannel`, ...) — there is no
precedent anywhere in this codebase for CRUD data that bypasses typiql.

## What one `#[typiql_type]` struct gives you for free

```rust
#[typiql_type]
pub struct ShakerChannel {
    #[typiql(key)]
    pub id: String,
    pub profile_id: Option<String>,
    pub pan: u8,
    pub devid: String,
    pub channels: u8,
    pub position: Option<String>,
}
```

Registered once in the `typiql_schema!(...)` list (`typiql_types.rs`), this
alone generates, with **zero hand-written resolver code**:

- `getShakerChannels(pan: Int, pan_ne: Int, pan_gt: Int, ..., profileId: String, profileId_ne: String, ...)` — one query, with filter arguments auto-generated **per field per applicable operator** (eq/ne/gt/gte/lt/lte/in/not_in, whichever make sense for that field's type). Filter server-side through these arguments — don't fetch everything and filter in application code when a filter argument already does it.
- `getShakerChannel(...)` — same filter args, returns the first match.
- `addShakerChannel(values: ShakerChannelInput!)`
- `updateShakerChannel(id: String!, update: ShakerChannelInput!)` — `ShakerChannelInput` fields are `MaybeUndefined<T>`: omitted = untouched, explicit `null` = clear, a value = set. Never write a custom "patch" resolver to get this semantic — it's already there.
- `removeShakerChannel(id: String!)`
- `shakerChannelChanged` subscription (add/update/remove events)

New IDs are auto-generated by the adapter (`Uuid::new_v4()`) whenever the key
field is absent from `add`'s input — never generate one yourself.

## Relations: prefer them over manual joins

A relation field resolves **one related record**, batched via a
`DataLoader<RelationLoader<T>>` across every row in the same GraphQL
selection (no N+1). Two forms, both already used in this codebase:

**Local/foreign field match** — most common, e.g. `Car.day_photo` resolving
against `File` by path (`typiql_types.rs`):

```rust
#[typiql(relation(local = "day_photo_path", op = "eq", foreign = "path"))]
pub day_photo: Option<File>,
pub day_photo_path: Option<String>,
```

**File/table relation** — for a "dumb" type with no default storage location
of its own, reached only through a parent (e.g. `DashboardEntry.dashboard`):

```rust
#[typiql(relation(file = "{path}/.dashboard.json", table = "dashboard"))]
pub dashboard: Option<Dashboard>,
```

Reach for a relation field whenever a resolver would otherwise need to fetch
a related record by matching one field against another — that's exactly
what `local`/`foreign` replaces, batched automatically instead of one
`get_one` call per row.

**Known gap**: relations currently only resolve to `Option<T>` (single
record) — there's no native one-to-many/`Vec<T>` relation. For "give me
every X where field == Y" (a fan-out, not a single lookup), use the
auto-generated list query's own filter argument (`getMonocoqueSoundDevices(pan: 3)`)
rather than writing a custom resolver to do the same filtering by hand.

## Custom resolvers: the narrow exception

A hand-written resolver (in `graphql/*.rs`, added to `ShakerDspMutation`/
`ShakerDspQuery`/etc. or a new `#[Object] impl`) is only appropriate for
functionality the auto-generated CRUD genuinely can't express — and normally
only when the user has specifically asked for that behavior, not as a
shortcut around writing the CRUD-driven equivalent client-side. Real
examples already in this codebase:

- `setDefaultSoundDeviceProfile` — enforcing "exactly one `is_default`" is a cross-row invariant the macro has no way to express.
- `applyShakerDspChannelLive`/`applyLfeChannelLive` — pushing a value to a **live PipeWire process** via `pw-cli`, not a CRUD operation on stored data at all.
- `seedMonocoqueSoundDevices` (internal, `api.rs`, not GraphQL-exposed) — parsing an *external* file (`monocoque.config`) to seed storage; parsing isn't CRUD.

**Not** a valid reason: "I need to create several related rows in one
round-trip." That's still just CRUD — do it the same way this codebase
already does it elsewhere, client-side, in sequence, using the
already-generated mutations (see `ShakerMatrix.tsx`'s `handleToggle` and
`Shakers/index.tsx`'s `cloneToProfile`, both of which call `add`/`remove`
multiple times in a loop rather than a bespoke atomic backend mutation). A
concrete mistake caught mid-session: writing `add_shaker_channel`/
`remove_shaker_channel` resolvers to atomically create a `ShakerChannel`
plus its six effect rows — wrong, since `ShakerChannel`/`MonocoqueSoundDevice`
CRUD is already fully auto-generated; that orchestration belongs in the
frontend exactly like every other multi-row creation flow in this app.

Even when a custom resolver *is* warranted, it must still operate through
the same typiql surface — `adapter.add/update/remove/get_many` against
`T::collection_name()`, returning/accepting the same auto-generated types —
never a parallel bespoke storage path or ad hoc JSON shape alongside what
the macro already generates.
